# 工厂模式(Factory Pattern)

将对象的创建过程封装起来，客户端不需要知道具体创建了哪个类的对象。

## 模式说明

工厂模式是最常用的创建型设计模式。根据抽象程度不同，分为三种：

1. **简单工厂模式**（Simple Factory）—— 不属于 GoF 23 种，但最基础
2. **工厂方法模式**（Factory Method）—— GoF 之一
3. **抽象工厂模式**（Abstract Factory）—— GoF 之一

三者共同点：都把对象的**创建和使用分离**，客户端面向接口编程。

---

## 一、简单工厂模式

### 模式定义

由一个工厂对象决定创建出哪一种产品类的实例。不属于 GoF 23 种设计模式，但是最基础的工厂实现。

### 角色划分

| 角色 | 说明 | 示例 |
|------|------|------|
| 抽象产品 | 定义产品的接口 | `Payment` |
| 具体产品 | 实现抽象产品接口 | `Alipay`、`WechatPay`、`BankCardPay` |
| 工厂类 | 包含创建产品的静态方法 | `SimplePaymentFactory` |

### 反面示例（直接在客户端 new 对象）

```java
// 客户端直接依赖具体类，耦合度高
public class Client {
    public void pay(String type, String orderId, BigDecimal amount) {
        if ("alipay".equals(type)) {
            new Alipay().pay(orderId, amount);         // 直接 new
        } else if ("wechat".equals(type)) {
            new WechatPay().pay(orderId, amount);      // 直接 new
        } else if ("bankcard".equals(type)) {
            new BankCardPay().pay(orderId, amount);    // 直接 new
        }
    }
}
```

**问题：** 客户端直接依赖具体类，对象创建逻辑散落在各处，难以维护。

### 正面示例（简单工厂）

```java
// 1. 产品接口
public interface Payment {
    void pay(String orderId, BigDecimal amount);
}

// 2. 具体产品
public class Alipay implements Payment {
    public void pay(String orderId, BigDecimal amount) {
        System.out.println("[支付宝] " + orderId + " 支付成功，金额: " + amount);
    }
}

// 3. 工厂类 —— 核心角色
public class SimplePaymentFactory {
    public static Payment create(String type) {
        return switch (type) {
            case "alipay"   -> new Alipay();
            case "wechat"   -> new WechatPay();
            case "bankcard" -> new BankCardPay();
            default -> throw new IllegalArgumentException("不支持的支付类型: " + type);
        };
    }
}

// 4. 客户端使用
Payment payment = SimplePaymentFactory.create("alipay");
payment.pay("ORDER_001", new BigDecimal("100.00"));
```

### 优点

- 客户端不需要知道具体产品类名，只需要传入类型参数
- 实现了对象创建和使用的分离

### 缺点

- **违反开闭原则** —— 新增产品需要修改工厂类的 switch 分支
- 工厂类职责过重，一旦不能工作，整个系统受影响

## 二、工厂方法模式

### 模式定义

定义一个创建对象的接口，让子类决定实例化哪一个类。工厂方法使一个类的实例化延迟到其子类。

### 角色划分

| 角色 | 说明 | 示例 |
|------|------|------|
| 抽象产品 | 定义产品的接口 | `Payment` |
| 具体产品 | 实现抽象产品接口 | `Alipay`、`WechatPay`、`BankCardPay` |
| 抽象工厂 | 声明工厂方法 | `PaymentFactory` |
| 具体工厂 | 实现工厂方法，创建具体产品 | `AlipayFactory`、`WechatPayFactory` |

### 正面示例

```java
// 1. 抽象工厂（工厂接口）
public interface PaymentFactory {
    Payment createPayment();  // 工厂方法，由子类决定创建什么
}

// 2. 具体工厂 —— 每种产品一个工厂
public class AlipayFactory implements PaymentFactory {
    public Payment createPayment() {
        return new Alipay();
    }
}

public class WechatPayFactory implements PaymentFactory {
    public Payment createPayment() {
        return new WechatPay();
    }
}

// 3. 客户端 —— 通过构造器注入工厂
public class PaymentStore {
    private final PaymentFactory factory;

    public PaymentStore(PaymentFactory factory) {
        this.factory = factory;
    }

    public void orderPayment(String orderId, BigDecimal amount) {
        Payment payment = factory.createPayment();  // 多态创建
        payment.pay(orderId, amount);
    }
}

// 4. 使用
PaymentStore store = new PaymentStore(new AlipayFactory());
store.orderPayment("ORDER_001", new BigDecimal("100.00"));
```

### 优点

- **符合开闭原则** —— 新增产品只需新增产品类 + 工厂类，无需修改已有代码
- **符合单一职责** —— 每个工厂只负责创建一种产品
- 客户端依赖抽象，不依赖具体实现

### 缺点

- 类的数量成对增加（每增加一个产品就要增加一个工厂）
- 增加了系统的抽象性和理解难度

## 三、抽象工厂模式

### 模式定义

提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们具体的类。

### 角色划分

| 角色 | 说明 | 示例 |
|------|------|------|
| 抽象产品A | 支付接口 | `Payment` |
| 抽象产品B | 回执接口 | `Receipt` |
| 具体产品 | 各产品族的实现 | `AlipayPayment`、`AlipayReceipt` |
| 抽象工厂 | 声明多个工厂方法 | `PaymentFactory` |
| 具体工厂 | 创建一族产品 | `AlipayFactory`、`WechatFactory` |

### 产品族概念

```
              Payment(支付)    Receipt(回执)
支付宝族 →    AlipayPayment    AlipayReceipt
微信族   →    WechatPayment    WechatReceipt
```

抽象工厂保证**同一族的产品能配合工作**。

### 正面示例

```java
// 1. 两个抽象产品
public interface Payment {
    void pay(String orderId, BigDecimal amount);
}
public interface Receipt {
    void generate(String orderId, BigDecimal amount);
}

// 2. 抽象工厂 —— 创建一族产品
public interface PaymentFactory {
    Payment createPayment();
    Receipt createReceipt();
}

// 3. 具体工厂 —— 支付宝族
public class AlipayFactory implements PaymentFactory {
    public Payment createPayment() { return new AlipayPayment(); }
    public Receipt createReceipt() { return new AlipayReceipt(); }
}

// 4. 具体工厂 —— 微信族
public class WechatFactory implements PaymentFactory {
    public Payment createPayment() { return new WechatPayment(); }
    public Receipt createReceipt() { return new WechatReceipt(); }
}

// 5. 使用
PaymentFactory factory = new AlipayFactory();
Payment payment = factory.createPayment();
Receipt receipt = factory.createReceipt();
payment.pay("ORDER_001", new BigDecimal("100.00"));
receipt.generate("ORDER_001", new BigDecimal("100.00"));
```

### 优点

- 保证同一产品族中的对象一起工作（风格一致）
- 客户端与具体产品的创建解耦
- 新增产品族只需新增工厂（符合开闭原则）

### 缺点

- **新增产品等级困难** —— 比如新增"退款(Refund)"产品，需要修改所有工厂接口和实现
- 系统更加复杂

---

## 三种工厂模式对比

| 维度 | 简单工厂 | 工厂方法 | 抽象工厂 |
|------|---------|---------|---------|
| 工厂个数 | 1 个 | 多个 | 多个 |
| 产品等级 | 单一 | 单一 | 多个（产品族） |
| 开闭原则 | ❌ 违反 | ✅ 遵守 | ⚠️ 部分遵守（新增族遵守，新增等级违反） |
| 抽象程度 | 低 | 中 | 高 |
| 适用场景 | 产品少且固定 | 产品经常扩展 | 需要创建产品族 |
